## 프로젝트 방향 구체화 및 공통 구조 설계

### 목적
- 게시판 주제와 MVP 범위 확정
- 구현을 위한 공통 설계 기준 마련

## 게시판 주제 선정
- 대분류 : 회원 기반 커뮤니티 기록 게시판
- 컨셉 : 사용자의 학습 내용, 문제 해결 경험 등을 기록하는 개인 위키형 지식 공유 게시판

### 주제 선정 이유
- AI의 발달로 학습 속도가 빨라졌지만, 얻은 지식이 휘발되기 쉽고 정보의 정확성을 개인이 판단하기 어려운 경우가 많다고 느꼈습니다.
- 학습 내용과 문제 해결 경험 등을 개인에 맞게 체계적으로 기록하고, 필요할 때 다시 꺼낼 수 있는 플랫폼이 필요하다고 생각했습니다.
- 단순히 개인 기록에 그치지 않고, 타인과 공유하며 검증할 수 있는 개인 위키형 게시판 서비스를 구현하고자 했습니다.

## 1차 MVP 범위 확정

### MVP
- MVP(Minimum Viable Product)란 서비스의 핵심 가치를 검증할 수 있는 최소 기능 제품입니다.
- **핵심 기능**을 먼저 정의하고, 이후 단계적으로 확장하기 위한 기준이 됩니다.

### MVP 범위를 먼저 정하는 이유
- MVP는 **서비스의 핵심 기능**을 최소 범위로 정의한 1차 구현 기준이 됩니다.
- 초기부터 많은 기능을 한 번에 구현하려고 하면 범위가 커지고, 그에 따라 핵심 기능의 완성도가 떨어질 수 있습니다.

### 1차 MVP 목표
- 회원 인증과 게시판의 핵심 CRUD 흐름을 구현합니다.
- 게시글, 댓글, 권한 처리, 입력값 검증, 예외 처리를 포함한 백엔드 기본 기능을 우선 완성합니다.

### 포함 기능
- Spring Boot 프로젝트 생성 및 환경 설정
- DB 연결 및 JPA 설정
- 공통 응답 형식 `ApiResponse<T>`
- 전역 예외 처리
- Swagger 설정
- JPA Auditing 설정
- 회원가입 / 로그인 / 내 정보 조회
- 게시글 작성 / 목록 조회 / 상세 조회 / 수정 / 삭제
- 댓글 작성 / 조회 / 수정 / 삭제
- 게시글 페이지네이션 및 검색
- 작성자 본인 여부 기반 권한 처리
- 기본 서비스 단위 테스트

## 패키지 구조 원칙 정하기
- 구현 과정에서 코드가 섞이지 않도록, 기능 영역과 공통 영역을 구분하는 패키지 구조 원칙을 먼저 정리하고자 했습니다.

### 기능 영역과 공통 영역
- 기능 영역은 회원, 게시글, 댓글처럼 서비스의 핵심 도메인별로 나누는 영역입니다.
- 공통 영역은 여러 기능에서 함께 사용하는 응답 형식, 예외 처리, 설정과 같은 요소를 모아두는 영역입니다.

### 패키지 구조 초안
- 기본적으로는 기능 영역과 공통 영역을 기준으로 나눕니다.
- 각 기능 영역 내부에서는 계층별로 코드를 분리합니다.
- 기능 영역은 `member`, `post`, `comment`와 같은 핵심 도메인을 기준으로 나눕니다.
- 각 기능 영역 내부에는 `controller`, `service`, `repository`, `dto`, `entity`와 같은 계층별 하위 패키지를 둡니다.
- 공통 영역은 별도의 `global` 패키지로 분리해서 공통 응답, 예외 처리, 설정 등을 관리합니다.

### 주요 계층
- `controller` : HTTP 요청을 받고 응답을 반환하는 역할
- `service` : 비즈니스 로직을 처리하는 역할
- `repository` : DB에 데이터를 저장하고 조회하는 역할
- `dto` : 요청값과 응답값을 전달하는 객체
- `entity` : DB 테이블과 매핑되는 객체

### 주요 도메인
- `member` : 회원가입, 로그인, 내 정보 조회 등 회원 기능을 담당하는 도메인
- `post` : 게시글 작성, 조회, 수정, 삭제와 목록 조회를 담당하는 도메인
- `comment` : 댓글 작성, 조회, 수정, 삭제를 담당하는 도메인

### 공통 영역
- `global` : 공통 응답, 예외 처리, 설정 등 여러 기능에서 함께 사용하는 요소를 관리하는 영역

## 공통 응답 형식 초안 설계
- API 응답 형식을 일관되게 유지하기 위해 공통 응답 형식을 먼저 정의하고자 했습니다.

### 공통 응답 형식의 필요성
- 응답 형식이 일관되지 않으면 클라이언트에서 처리하기 어렵고, API 문서화 및 유지보수도 복잡해질 수 있습니다.
- 성공 응답과 실패 응답의 기본 구조를 통일하면, 기능이 늘어나더라도 일관된 방식으로 응답을 관리할 수 있습니다.

### 응답 형식 초안
- 공통 응답 형식은 `success`, `message`, `data` 필드를 가지는 구조입니다.
- `success`는 요청 성공 여부를 나타냅니다.
- `message`는 응답 결과에 대한 설명을 담습니다.
- `data`는 실제 응답 데이터를 담는 필드로 사용합니다.
- 성공 응답과 실패 응답 모두 기본적으로 같은 틀을 유지하고, 필요한 경우 에러 코드를 추가로 확장할 수 있게 고려했습니다.

```json
{
  "success": true,
  "message": "게시글 조회 성공",
  "data": {
    "id": 1,
    "title": "제목"
  }
}
```

## 예외 처리 구조 초안 설계
- 예외 상황이 발생했을 때 일관된 방식으로 응답하기 위해 예외 처리 구조를 먼저 정리하고자 했습니다.

### 예외 처리 구조의 필요성
- 기능이 많아질수록 다양한 예외 상황이 발생할 수 있기 때문에, 예외를 일관된 방식으로 처리할 기준이 필요합니다.
- 예외 처리 기준이 없으면 컨트롤러나 서비스마다 서로 다른 형태의 에러 응답이 내려갈 수 있어 유지보수가 어려워질 수 있습니다.
- 공통된 예외 처리 구조를 두면, 클라이언트에서도 일정한 응답 형식을 기준으로 에러 상황에 대응할 수 있습니다.

### 예외 처리 구조 초안
- 예외 처리는 공통된 에러 코드와 사용자 정의 예외, 전역 예외 처리기를 중심으로 구성하는 방향을 고려했습니다.
- 에러 상황은 `ErrorCode`를 기준으로 분류하고, 비즈니스 로직에서 발생하는 예외는 사용자 정의 예외로 관리합니다.
- 최종적으로는 `GlobalExceptionHandler`를 통해 예외를 한 곳에서 처리하고, 일관된 에러 응답 형식으로 반환하는 구조를 목표로 했습니다.

### 주요 구성 요소
- `ErrorCode`는 예외 상황을 구분하기 위한 코드입니다.
  - 예) `POST_NOT_FOUND`, `INVALID_INPUT`, `UNAUTHORIZED`
- `BusinessException`은 비즈니스 로직에서 발생하는 예외를 직접 정의하여 사용하는 사용자 정의 예외입니다.
  - 예) 게시글이 없을 경우 -> `throw new BusinessException(ErrorCode.POST_NOT_FOUND)`
- `GlobalExceptionHandler`는 발생한 예외를 한 곳에서 받아 일관된 응답 형식으로 반환하는 역할을 합니다.
