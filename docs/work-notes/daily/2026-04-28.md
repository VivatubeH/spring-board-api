## 회원 도메인 설계

### 목적
- 회원 엔티티, 역할 구조, 공통 이력 관리 방향을 정리해서 회원가입과 로그인 기능 구현의 기반을 마련한다.

## 회원가입/로그인에 필요한 필드 정리

### 회원 필드 정리가 필요한 이유
- 회원가입과 로그인 기능을 구현하려면, 회원이 어떤 정보를 가져야 하는지 정리할 필요가 있다.
- 필드를 먼저 정리해두면 이후 엔티티 설계와 기능 구현 방향을 더 명확하게 잡을 수 있다.

### 필드란?
- 필드는 엔티티 안에 저장되는 하나의 데이터 항목을 의미한다.
- 회원 엔티티에서는 이메일, 비밀번호, 닉네임 등이 필드에 해당한다.

### 회원가입에 필요한 필드
- 회원가입 기능을 위해서는 사용자를 식별하고 인증할 수 있는 기본 정보가 필요하다.
- `email` : 회원을 식별하고 로그인에 사용할 정보
- `password` : 회원 인증에 사용할 비밀번호
- `nickname` : 서비스 내에서 표시할 이름
- 로그인 식별자는 별도 아이디 대신 이메일을 사용한다.

### 로그인에 필요한 필드
- 로그인 기능에는 회원을 식별할 정보와 인증을 위한 비밀번호가 필요하다.
- `email` : 로그인 식별자 역할을 하는 정보
- `password` : 회원 인증에 사용할 비밀번호

### 공통으로 관리할 필드
- 회원 정보에는 회원가입과 로그인에 직접 사용되는 값 외에도, 서비스 전반에서 공통으로 관리해야 하는 정보가 필요하다.
- 현재 단계에서는 이메일, 비밀번호, 닉네임과 함께 역할 정보와 공통 이력을 함께 관리하는 방향을 고려한다.
- `email` : 회원 식별 로그인에 사용하는 정보
- `password` : 회원 인증에 사용하는 정보
- `nickname` : 서비스 내 표시되는 이름
- `role` : 회원 권한 구분 정보 (예: 일반 회원인지 관리자인지)
- 생성 시각 / 수정 시각 : 공통 이력 관리 정보 (회원 정보가 언제 처음 만들어졌고, 언제 마지막으로 수정되었는지)

### 필드 선정 기준
- 회원가입과 로그인 기능 구현에 직접 필요한 정보인지 최소한으로 고려했다.
- 회원을 식별하고 인증하는 필요한 정보인지 기준으로 삼았다.
- 이후 권한 관리와 같은 확장 가능성을 고려해 역할 정보를 포함했다.
- 데이터의 생성 시점과 수정 시점을 추적할 수 있도록 공통 이력 정보도 함께 포함했다.
- 1차 MVP에서는 과도한 필드 추가는 피하고, 핵심 기능 구현에 필요한 최소 범위를 우선 반영한다.

### 1차 MVP에서 사용할 회원 필드 초안
- `email` : 회원 식별, 로그인에 사용할 정보
- `password` : 회원 인증에 사용할 비밀번호
- `nickname` : 서비스 . 표시 이름
- `role` : 회원 권한 구분 정보
- 생성 시각 / 수정 시각 : 공통 이력 관리 정보

## 역할(Role) 구조 초안 작성

### Role이란?
- Role은 회원의 권한 수준을 구분하기 위한 정보이다.
- 서비스에서는 이 값을 기준으로 회원이 사용할 수 있는 기능 범위를 나눌 수 있다.

### Role이 필요한 이유
- 서비스에서 회원의 권한 범위를 구분하기 위해 Role 정보가 필요하다.
- 이를 통해 이후 기능별 접근 제어를 적용할 수 있는 기반을 마련할 수 있다.

### 1차 MVP에서 고려할 역할 구분
- 1차 MVP 단계에서는 우선 일반 회원과 관리자 역할을 구분하는 수준으로 정리한다.
- `USER`는 기본 회원 권한을 의미하고, `ADMIN`은 관리 기능까지 포함할 수 있는 확장 역할로 고려한다.

### Role 적용 방향
- 회원 권한은 회원 엔티티의 필드로 함께 관리하는 방향으로 설계하고자 했다.
- 권한 값은 문자열보다는 명확한 구분이 가능하도록 `enum`으로 관리하는 방식을 고려한다.

## 회원 엔티티 설계

### 회원 엔티티 설계란?
- 회원 엔티티 설계는 앞에서 정리한 회원 필드와 역할 정보를 실제 회원 객체 구조로 묶어 정리하는 과정이다.
- 즉, 회원이 어떤 데이터를 가지고 어떻게 관리될 지를 하나의 엔티티 기준으로 구체화하는 단계이다.

### 회원 엔티티 설계가 필요한 이유
- 회원 정보를 저장하고 관리하기 위한 기준 구조를 먼저 정리하기 위해 회원 엔티티 설계가 필요하다.
- 이를 통해 이후 JPA 매핑과 기능 구현 방향을 더 명확하게 잡을 수 있다.

### 회원 엔티티에 포함할 필드
- `email` : 회원 식별 및 로그인 정보
- `password` : 회원 인증 정보
- `nickname` : 서비스 내 표시 이름
- `role` : 회원 권한 구분 정보
- 생성 시각 / 수정 시각 : 공통 이력 관리 정보
- 주소, 주민등록번호, 휴대폰 번호 같은 추가 개인정보도 회원 엔티티에 포함될 수 있지만, 1차 MVP에서는 회원 식별과 인증, 권한 관리에 필요한 최소 필드만 우선 반영한다.

### 회원 엔티티 설계 방향
- 회원 엔티티는 식별, 인증, 권한 관리에 필요한 필드를 중심으로 구성한다.
- 1차 MVP에서는 최소 필드만 우선 반영하고, 공통 이력 정보는 이후 Auditing과 연결할 수 있도록 고려한다.

## Auditing 적용 방향 연결

### 회원 엔티티와 Auditing의 관계
- 회원 엔티티에는 생성 시각과 수정 시각과 같은 공통 이력 정보가 포함될 수 있다.
- 이러한 값은 JPA Auditing을 통해 자동으로 관리할 수 있다.

### 회원 엔티티에 적용할 공통 이력 값
- 회원 엔티티에는 생성 시각과 수정 시각을 공통 이력 값으로 포함하는 방향을 고려한다.
- 이러한 공통 이력 값은 여러 엔티티에서 함께 사용할 수 있다.

### Auditing 적용 방향 정리
- 생성 시각과 수정 시각은 회원, 게시글, 댓글 엔티티에서 공통으로 필요한 이력 정보로 본다.
- 이후 JPA Auditing을 적용하여 이러한 공통 값을 여러 엔티티에서 일관되게 관리할 수 있도록 구성할 예정이다.
